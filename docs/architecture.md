# 架构说明

## 设计原则

- 单一职责：状态机、渲染、终端生命周期、易经算法、随机取熵分别落在独立模块
- 可测试：把“取熵”抽象成 `EntropySource` trait，使状态机可以用脚本化输入测试
- 可扩展：`yijing` 模块不依赖 TUI，可以独立复用到别的前端

## 依赖方向

```text
main
 ├─ tui
 └─ app
    ├─ ui
    ├─ entropy
    └─ yijing
```

约束：

- `ui` 不直接生成随机数
- `yijing` 不依赖终端和事件系统
- `entropy` 不接触 UI 状态

## 状态职责

### `Welcome`

- 展示仪式入口提示
- 首次 `Enter` 后立即进入 `Casting`

### `Casting`

- 每次 `Enter` 只生成一枚 `2/3`
- 每满 3 次生成一条爻
- 第 18 次后切到 `Assembling`

### `Assembling`

- 从“基础数字”视角切到“六爻骨架”视角
- 等待用户确认进入倒序显示

### `ReverseConfirm`

- 明确提示“倒转乾坤”
- 用户确认后进入最终结果页

### `Result`

- 渲染六爻图、本卦名、变爻说明
- 再按 `Enter` 会重新开始新一轮起卦

## 可维护扩展点

### 1. 变卦

当前已经输出本卦、变爻列表和之卦：

- `6 -> 7`
- `9 -> 8`
- `7/8` 保持不变

变卦推演逻辑集中在 `yijing.rs`，UI 只消费结果。

### 2. 卦辞/爻辞

建议不要直接塞进 `App`。

更好的做法是：

- 在 `yijing.rs` 新增静态数据库结构
- 让 `HexagramResult` 增加可选字段
- UI 按需渲染，不改变状态机

### 3. 动画与视觉

如果需要更强的“仪式感”：

- 在 `ui.rs` 内增加闪烁、颜色渐变、段落切页
- 保持 `app.rs` 不承担任何渲染细节

## 测试策略

当前已有两类基础测试：

- `entropy.rs`
  验证即时随机结果始终落在 `2/3`
- `yijing.rs`
  验证典型卦名映射、变爻格式和爻描述
- `app.rs`
  用脚本化熵源走完整个状态机流程
- `ui.rs`
  用 `TestBackend` 做欢迎页和结果页快照测试

后续建议补充：

- 不同六爻组合的参数化测试
- 小终端尺寸下的 UI 快照测试
- 重新起卦时状态清理是否完整的测试
